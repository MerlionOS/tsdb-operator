## 引子：2013 年那条岔路

2013 年。有两件事同时在发生。

在柏林，Julius Volz 和 Matt Proud 的 Prometheus 已经在 SoundCloud
跑了大半年，开始吸引第一批外部贡献者。

在北京，滴滴出行的运维团队正在内部开发一个叫**夜莺**的监控告警
平台。它要解决的问题和 Prometheus 相近，但思路不同 —— Prometheus
是工程师自己写给自己用的工具链；夜莺是一个**面向运维管理员**的
产品。

同一年，一家叫 **PagerDuty** 的美国 SaaS 在 IPO 的路上飞奔。而
在中国，**没有任何一家公司在认真做它的对标产品** —— 国内团队用
钉钉群、电话机器人、短信网关自己拼凑。

那时候没有人想到，十年之后，围绕 Prometheus 这套核心技术，东方
和西方会长出**两套截然不同的可观测性生态**。

但如果你读过卷一，你会觉得这件事**似曾相识**。

---

## 第一章 · 现代分叉的三个真问题

### 第一问：为什么 PagerDuty 进不了中国？

PagerDuty 是北美事件响应的默认选项。但在中国，它几乎不存在。

原因你我都懂：

- 国际信用卡门槛
- 数据出境和合规限制
- 钉钉、企业微信、飞书集成缺位 —— **工作不在 Slack 里发生**

空白的市场必然有人填。于是出现了：

- **Flashduty**（快猫星云）—— 对标 PagerDuty 的商业产品
- **OpsPilot**、**极狐 IncidentOps** —— 类似定位

有意思的是，这些产品**不是简单抄**。

PagerDuty 原版的告警降噪能力其实一般，国内产品普遍把"相同指
标、相同服务、相近时间窗"的告警聚合得**更狠**。PagerDuty 和
Slack 集成做得很好，国内产品则和钉钉机器人、企业微信群、飞书
卡片做了**交互式响应** —— 在 IM 里直接确认告警、转派、升级、
关闭。这个交互密度 PagerDuty 做不到。

国内 IncidentOps 产品也更早上 AI —— 告警摘要、根因推荐、事件
时间线 LLM 总结。效果未必比硅谷强，但**敢上**。

为什么敢上？因为国内企业软件的销售路径允许"半成品先进去"。硅
谷 SaaS 更在意 NPS 和 churn，AI 功能要等到真有用才发。

**这不是技术差异，是组织差异。**

### 第二问：为什么 Pixie 凉了，而 DeepFlow 升温？

2018 年。加州。一家叫 **Pixie Labs** 的初创开始做 eBPF 应用观
测。它的思路很先进 —— 用 eBPF 在 Linux 内核里拦截应用行为，零
侵入，自动生成 metrics、traces、logs。

Pixie 在早期社区里被寄予厚望，被称为"下一代 APM 的种子"。

2020 年，Pixie 被 New Relic 收购。

收购之后，Pixie 慢慢淡出。核心团队陆续离开，社区贡献放缓，发
布节奏从"每两周一版"变成"每半年一版"。到 2023 年，Pixie 实
际上已经停摆。

同一段时间，**DeepFlow**（云杉网络开源）在 CNCF Landscape 里慢
慢爬上来。2023 年进了 CNCF Sandbox，Stars 越过 7000，贡献者从
国内扩展到东南亚和欧洲。

**赛道相同，路径相反。**

Pixie 走的是"被大厂收购之后消失"的路径 —— 这在硅谷是很多开源
创业公司的默认结局。DeepFlow 走的是"独立创业 + 开源核心 + 商
业增值"的路径。

这里有一个国内市场的结构性特征：**大厂收购开源项目的文化没有
硅谷浓**。原因复杂 —— 开源估值模型不成熟、收购整合成本高、
**国内用户对大厂垄断开源更警惕**。这让国内的可观测性创业公司
有一个"死不掉也卖不掉"的生存区间。

Pixie 死了，但 eBPF 可观测性这个方向没死。它**在国内活下来
了**，并且在全球范围内都在填 Pixie 留下的空位。

同赛道还有 **Kindling**（谐云），方向类似，更轻量一些。

这一块**国内没有在追，而是在领**。

### 第三问：为什么 Datadog 的商业模式在国内活不下来？

硅谷一家 C 轮公司每年付 15 万美元给 Datadog，工程师咬牙也要
掏。因为这笔钱换的是"我们不用自己维护监控栈"的时间。

国内一家同规模公司，这笔钱可能是**全公司一整年的服务器预算**。

所以国内团队对"每块钱能存多少 metrics"非常敏感。

- **VictoriaMetrics** 在国内的渗透率远高于硅谷。它单节点每个
  sample 的压缩密度能做到 0.4 字节（Prometheus 默认 1.3 字节）。
  对预算敏感的团队，这是**结构性优势**
- **TDengine**（涛思数据）押 IoT 和工业时序 —— 国内制造业、
  智能设备、新能源车的时序数据量级巨大，Prometheus + VM 撑不
  住
- **GreptimeDB** 赌的是 metrics + logs + traces 一体化，Rust +
  Arrow，云原生。押的是一条硅谷大厂**不太愿意下场做的赛道**
  （因为对 Datadog、Grafana 都是自我革命）

还有一个被低估的因素：**等保**（网络安全等级保护）。

等保对可观测性的审计日志、操作记录、数据留存有**非常具体**的
要求。这在硅谷没有对应的监管场景。所以国内可观测性产品里，
**审计是一等公民**，而硅谷产品经常是靠集成第三方日志系统来
应付。

## 第二章 · 其实我们三千年前就分叉过

如果你读过卷一，你可能已经开始觉得上面三个问题**似曾相识**。

**第一问**（PagerDuty 进不来，IncidentOps 本土化）—— 国内产品
深度集成到管理层、面向汇报、合规导向。产品长得像**钦天监**：
中央机构、层级清晰、对决策者交付。

**第二问**（Pixie 凉了，DeepFlow 升温）—— 前者的结局是**被资
本整合后消失**，后者的生存是**开源社区 + 商业化混合**。硅谷走
收购整合的路，国内走独立生态的路。

**第三问**（Datadog 不香，VM / TDengine / GreptimeDB 崛起）——
国内方案的底色是**低成本、高密度、一体化**，和中国制造业"又好
又便宜"的基因是同一个。

这三个现代分叉的底层模式，在卷一里已经出现过对应物：

| 古代 | 现代 |
|------|------|
| 钦天监 —— 集权、专业、对决策层汇报 | 夜莺、Flashduty、阿里云 ARMS —— 面向管理、一体化、合规 |
| Panopticon —— 观测即权力，观察者隐匿 | Datadog、New Relic —— SaaS 黑盒，数据被一家公司看着 |
| 驿站系统 —— 国家级分布式消息 | 钉钉、企业微信 —— 组织级 IM + 告警通道 |
| Paul Revere 的灯笼 —— 个体的、草根的告警 | Grafana、Prometheus —— 工程师社区长出来的工具 |
| Nightingale 的玫瑰图 —— 一个护士用数据改制度 | Torkel Ödegaard 一个人 fork 出的 Grafana |
| 罗马 Census —— 周期性全量扫描 | OTel 语义约定 —— 跨厂商的标准化 schema |

**东方路径的基因：自上而下、集权、汇报导向、合规一等公民。**
**西方路径的基因：自下而上、分散、个体驱动、标准导向。**

三千年后，这两条路径还在**各自演化**。

这不是谁对谁错。这是两种完全不同的**组织环境**长出的两种完全
不同的生态。

中国的可观测性产品显得"管理味重"，不是因为工程师不懂工具，是
因为这些工具在**上级对下级问责文化更强**的组织里被使用。硅谷
的可观测性产品显得"工具味重"，不是因为管理层没权威，是因为
这些工具在**工程师自主权更大**的组织里被使用。

**现代的分叉，是古代基因的回响。**

## 第三章 · 国内并不是处处在领

公平地说一句。

**AI Copilot 方向，国内在追。**

Datadog Bits、Grafana Asserts、Elastic AI Assistant —— 硅谷的
头部产品已经做到"你问它'最近为什么 p99 飙高'能给出合理猜
测"的程度。国内产品大多还停在"告警摘要帮你写一段话"的阶段。

这不是能力问题，是**数据和上下文的问题**。Datadog 能做这些是
因为它有几万家客户的跨产品上下文作为训练语料。国内产品还在积
累这个语料。

**OpenTelemetry 生态参与度，国内在追。**

OTel 的 spec 是全球可观测性的事实标准。核心贡献者、SIG chair、
语义约定设计者，大部分来自硅谷和欧洲的大厂 —— Microsoft、
Google、Splunk、New Relic。国内公司的参与度在涨，但离能左右
spec 的程度还远。

这个会**吃亏**。OTel 的 semantic conventions 每一条都在定义未
来几年全球采集的"方言"。如果国内场景（比如等保审计字段、
中文 IM 结构化消息）没进 spec，后面就得自己做转换层。

**大规模长期存储的工程成熟度，国内在追。**

Thanos、Mimir 在硅谷大厂的生产案例密度还是更高。国内团队做
大规模长期存储的经验公开案例并不多 —— 一部分是分享文化不同，
一部分是**规模真的没到那个量级**。

## 第四章 · 那选型到底该看什么？

绕了一圈回到实际问题：一个 2026 年的团队要做可观测性选型，该
怎么看？

**三条决策线，按顺序过：**

### 决策线一：业务形态决定骨架

- **SaaS / 互联网** → Prometheus + VM / Thanos + 夜莺 / Grafana
- **IoT / 工业 / 车联网** → 严肃考虑 TDengine，别硬塞 Prometheus
- **混合云 / 多云** → 谨慎选云厂商强绑定的托管，迁移成本两年
  后打脸
- **金融 / 合规敏感** → 自建 + 审计 + 等保合规，基本没得选

### 决策线二：预算决定要不要上托管

- **< 100 人工程团队**：自建成本比托管高，但国内托管选项有限，
  实际往往是"开源自建 + 运维兼职"
- **100–500 人**：混合路线最常见。核心业务自建，边缘集群托管
- **> 500 人**：基本都是自建 + 专职 SRE。托管只作为补充

### 决策线三：团队规模决定自建深度

- **10 人以下**：别自建 Thanos / Mimir，跑不动。单机 Prometheus
  + 备份 + 托管告警，够用
- **10–50 人**：可以上 VictoriaMetrics 集群，但要有一个**懂运
  维的人专门兜底**
- **50 人以上**：才真有本钱玩 Thanos / 多 region / 跨集群聚合

一句话：**可观测性栈的复杂度，不该超过团队能运维它的上限。**

国内很多团队踩的坑，是"按头部公司架构抄作业"，结果三个人维护
不起来，半夜一个告警一响，整个系统瘫一天。

## 第五章 · 私货 —— 我写的那个小小"钦天监"

写到这里，该交代一下我自己的位置了。

过去两年，我帮几家公司看过 Prometheus 自建栈，看到一个重复到
让我想吐的现象：**每家公司都在重写同一套脚手架**。

- 拉 StatefulSet、挂 PVC、写 headless Service、配 ConfigMap
- 写 cron 脚本调 Prometheus admin snapshot，`tar` 打包，
  `aws s3 cp` 上传
- 健康检查，坏副本得手动 `kubectl delete pod` 重建
- 变更记录？K8s Event 一小时后就查不到
- 跨 namespace 聚合？没有，自己写脚本扫

这些东西不是什么技术难点。**它们就是琐碎且容易漏掉一个角。**

于是我把它们抽出来，封装成了一个 Kubernetes Operator。名字叫
**`tsdb-operator`**。

它做的事情：

- 一个 CRD `PrometheusCluster` —— StatefulSet + PVC + Service +
  ConfigMap 一次性生好
- **自动定期快照到 S3 / MinIO**，配套 `tsdb-ctl` 命令行工具做
  list 和 restore
- 副本健康检查 + 自动剔除重建
- **PostgreSQL 审计日志** —— 每次集群变更、每次备份事件都落库，
  带 retention 和 pruner。这一条是国内等保场景倒逼我加的
- 跨 namespace 聚合的 `PrometheusClusterSet`（cluster-scoped CRD）
- 可选的 gin REST API 和 Grafana 面板

它故意做得**很窄**：

- 不做告警 UI（那是夜莺、Grafana 的事）
- 不做 APM（那是 SkyWalking 的事）
- 不做采集层（那是 DeepFlow 的事）
- 不做 TSDB 本身（那是 Prometheus / VictoriaMetrics / GreptimeDB
  的事）

它只做一件事：**在 Kubernetes 上把 Prometheus 集群跑稳、备好
份、审计到位。**

某种意义上，它是一个**小小的现代钦天监** —— 专职盯着一组
Prometheus 集群别挂，每一次变更都有记录（像起居注那样），每一
次快照都落到对象存储（像驿站把文书抄送四方那样）。

它和本文提到的大多数国产方案，**默认关系是互补，不是竞争**：

- 夜莺要一个 metrics 后端 → tsdb-operator 把那个后端跑稳
- DeepFlow 要 remote-write metrics → 写到 tsdb-operator 管的
  Prometheus
- 想把长期数据放到 VictoriaMetrics / GreptimeDB → 配 remote-write
  就行，tsdb-operator 不碰那一层

如果你正好在国内团队自建 Prometheus，又嫌那些脚手架烦，可以瞥
一眼：

- GitHub：`MerlionOS/tsdb-operator`
- 项目里有一篇[《中国大陆可观测性方案横向速览》](https://github.com/MerlionOS/tsdb-operator/blob/main/docs/CHINA-LANDSCAPE.zh.md)，
  是本文技术版

欢迎提 issue。欢迎骂。**骂得有道理的我改。**

## 史诗终章 · 回到骊山

写到这里，故事该绕回开头了。

**公元前 771 年。骊山。**

周幽王为了博褒姒一笑，点燃了烽火。诸侯星夜赶到，只看到一个女人
在笑。三年后，犬戎真的来了，烽火再起，无人相信。西周亡。

**公元 2026 年。北京 / 杭州 / 深圳 / 旧金山 / 柏林的某个工位
上。**

一个工程师盯着屏幕上 Slack 告警群炸出的第十五条消息，用冷漠
的手指按下 "mute this alert"。告警没有停。但它已经没有意义了。

三小时后，线上真的挂了。

---

工具变了三千年。

铜到电。烽火到 Prometheus。灯笼到 PagerDuty。浑天仪到 Grafana。

**但信息不对称的焦虑，一点没变。**

**告警疲劳的机制，一点没变。**

**集权观察与个体观察的张力，一点没变。**

**观察者如何避免污染被观察对象的问题，一点没变。**

你今天做可观测性选型，不只是在挑一个技术栈。你是在给你的组织，
在三千年人类观察史的坐标系里，**选一个位置**。

选钦天监的路径，你会得到一个对中央决策层友好的、合规一等公民
的、层级清晰的系统 —— 代价是工程师创新的自由度。

选 Panopticon 的路径，你会得到一个工具驱动的、标准化的、工程师
主导的系统 —— 代价是管理层汇报链条更难做。

选一条完全自建的路径，你就是在骊山点烽火的那个人 —— 自由地
制造告警，也自由地承担告警被忽略的代价。

选一条完全托管的路径，你就是被 Panopticon 观察的囚徒 —— 不知
道黑盒里的 SaaS 是怎么看你的，只能默认它在看。

**没有正确答案。只有你愿意付的代价。**

---

这是可观测性真正的史诗。

三千年来，它只关于一件事：

**在你看不见的地方出事之前，让你知道。**

从骊山山顶那一缕烟，到你手机上下一条告警推送 —— 这件事的本
质，没有一分钟变过。

---

**Fin.**
